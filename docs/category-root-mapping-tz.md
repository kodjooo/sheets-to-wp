# ТЗ: Жёсткая привязка категорий к корневым веткам (EN/PT)

## Цель
Исключить смешивание категорий из разных деревьев WooCommerce/WPML (например, `running` и `running-2`) и обеспечить стабильное назначение категорий для EN/PT товаров.

## Проблема
Сейчас родительская категория ищется по имени среди `parent=0`, из-за чего при наличии дублей/старых веток может выбираться неверный root. В результате:
- товар получает категории из разных веток;
- EN/PT категории выглядят несогласованно;
- при повторных прогонах возникают расхождения в итоговом наборе категорий.

## Требуемая доработка

### 1) Источник истины для root-категорий
Добавить конфигурацию маппинга корневых категорий (EN):
- ключ: текст из колонки `CATEGORY` таблицы (нормализованно);
- значение: фиксированный `root category id`.

Формат конфигурации:
- через env-переменную JSON (предпочтительно), либо отдельный конфиг-файл.

Пример:
```json
{
  "running": 4186,
  "cycling": 123,
  "swimming": 456,
  "triathlon": 789
}
```

### 2) Логика назначения категорий EN
Для EN товара:
1. Не выполнять свободный поиск root по имени.
2. Брать root строго из `root_category_map`.
3. `SUBCATEGORY` искать/создавать только внутри этого `parent_id`.
4. Если root для `CATEGORY` не найден в map — логировать ошибку и не назначать категории вне whitelist.

### 3) Логика назначения категорий PT
Для PT товара:
1. Взять EN root/category ids (из шага EN).
2. Получить PT IDs через WPML `translations["pt"]`.
3. Назначить PT категории только из этих translated IDs.
4. Если перевода нет — логировать и применять fallback по правилам (см. ниже).

### 4) Fallback-правила (минимально допустимые)
- Если нет root в map: статус/лог ошибки, без поиска по всем деревьям.
- Если нет PT-translation у найденной EN категории: пропустить конкретную категорию и зафиксировать warning.
- Запрещено свободно добавлять root из других веток.

### 5) Нормализация сравнения CATEGORY
Для ключа map использовать нормализацию:
- trim;
- lowercase;
- схлопывание повторных пробелов.

Регистр букв (`Running`/`running`) не должен влиять на результат.

## Нефункциональные требования
- Логирование: явные записи о выбранном root ID, найденном/созданном subcategory ID, translation PT ID.
- Не менять текущую бизнес-логику статусов (`Revised/Published`) и вариаций.

## Критерии приёмки
1. Для одного и того же `CATEGORY/SUBCATEGORY` во всех прогонах назначается один и тот же root-ветка.
2. Товар не получает категории из нескольких параллельных деревьев.
3. EN и PT версии получают корректные категории своей языковой ветки.
4. При отсутствии map/translation нет тихого fallback в «чужие» ветки.

## Тест-кейсы
1. `CATEGORY=Running`, `SUBCATEGORY=Walking`:
- EN: root строго `running-2` (ID из map), child внутри него;
- PT: translated root/child от EN IDs.
2. Дубликат root с тем же именем в другой ветке:
- не используется.
3. Нет ключа в map:
- фиксируется ошибка/предупреждение, категории не назначаются из случайного дерева.
4. Нет PT-перевода child:
- warning, child пропускается, root остаётся корректным.
