Техническое задание
Проект: sheets-to-wp-racefinder
Задача: автоматическое восстановление и заполнение WP PRODUCT ID EN/PT и WP VARIATION ID EN/PT для ранее опубликованных товаров и вариаций WooCommerce

1. Цель

Реализовать отдельный технический сценарий восстановления идентификаторов WooCommerce для уже существующих опубликованных карточек событий и их вариаций, если эти идентификаторы ранее не были записаны в Google Sheets.

Сценарий должен позволять:
- автоматически находить и записывать WP PRODUCT ID EN;
- автоматически находить и записывать WP PRODUCT ID PT;
- автоматически находить и записывать WP VARIATION ID EN;
- автоматически находить и записывать WP VARIATION ID PT;
- минимизировать риск ошибочного сопоставления за счет приоритета надежных источников и явной обработки неоднозначных случаев.

Результат нужен для того, чтобы старые записи, опубликованные до внедрения полной записи ID, могли корректно обновляться через существующий pipeline со статусом Revised (complete).

2. Контекст и ограничения

- В проекте уже существует рабочий pipeline публикации и обновления товаров/вариаций WooCommerce.
- Обновление товаров выполняется по колонкам:
  - WP PRODUCT ID EN
  - WP PRODUCT ID PT
- Обновление вариаций выполняется по колонкам:
  - WP VARIATION ID EN
  - WP VARIATION ID PT
- Для старых записей эти поля могут быть пустыми.
- Названия товаров и значения PT-атрибутов могли меняться, поэтому простое сопоставление по текстовому имени товара является недостаточно надежным.
- Сайт использует WPML, WooCommerce, публичный WP REST API и WC Store API.

3. Формат реализации

Нужно реализовать отдельный recovery-инструмент, не смешивая его с основным ежедневным pipeline.

Допустимые варианты запуска:
- отдельный Python-скрипт в папке run/;
- либо отдельная функция/модуль, вызываемая вручную;
- предпочтительно отдельный исполняемый сценарий с понятным именем, например:
  - run/recover_wp_ids.py

Скрипт должен запускаться вручную и не должен автоматически стартовать внутри ежедневного расписания.

4. Основная задача

Для каждой главной строки события в Google Sheets, у которой отсутствует хотя бы одно из полей:
- WP PRODUCT ID EN
- WP PRODUCT ID PT
- WP VARIATION ID EN
- WP VARIATION ID PT

необходимо попытаться автоматически восстановить недостающие значения и записать их обратно в таблицу.

5. Какие строки нужно обрабатывать

5.1. Обрабатывать только главные строки событий:
- строки, у которых STATUS содержит один из рабочих статусов верхнего уровня:
  - Published
  - Published (incomplete)
  - Revised (complete)
  - Revised (incomplete)
- либо иной явно согласованный набор главных строк, если в текущей структуре таблицы главная строка определяется не только по STATUS.

5.2. Строки вариаций:
- дочерние строки без верхнеуровневого STATUS;
- должны обрабатываться только в связке с уже найденным основным товаром EN/PT.

6. Какие поля таблицы используются как входные данные

Для поиска и сопоставления использовать следующие поля, если они заполнены:

Основные поля товара:
- ID
- STATUS
- RACE NAME
- RACE NAME (PT)
- WEBSITE
- LINK RACEFINDER
- EVENT START DATE
- EVENT END DATE
- LOCATION
- LOCATION (CITY)
- CATEGORY
- SUBCATEGORY
- WP PRODUCT ID EN
- WP PRODUCT ID PT

Поля вариаций:
- ATTRIBUTE
- VALUE
- DISTANCE
- TEAM
- TYPE
- LICENSE
- RACE START DATE
- RACE START TIME
- PRICE
- WP VARIATION ID EN
- WP VARIATION ID PT

7. Источники данных для поиска в WordPress/WooCommerce

7.1. Приоритетные источники

Поиск идентификаторов должен идти по каскаду надежности.

Приоритет 1. Прямой ID из LINK RACEFINDER
- Если в LINK RACEFINDER присутствует URL вида:
  - ?p=26252
  - ?post=26252
  - /wp-admin/post.php?post=26252
то ID должен извлекаться напрямую.
- Это самый надежный способ восстановления ID.

Приоритет 2. Канонический product ID из публичной страницы товара
- Если есть публичный URL EN или PT, нужно:
  - загрузить HTML;
  - найти shortlink;
  - либо найти alternate JSON URL вида /wp-json/wp/v2/product/{id};
  - либо извлечь product ID из служебных ссылок страницы.

Приоритет 3. Определение PT-перевода через WPML-связь страницы
- Если известен EN URL или EN product ID, нужно попытаться найти PT-перевод:
  - через rel="alternate" hreflang="pt-pt";
  - через PT JSON product URL на PT-странице;
  - через иные доступные публичные признаки WPML.

Приоритет 4. Поиск по WEBSITE как внешнему ключу
- У товара в ACF сохраняется event_ticket_url.
- Если известен WEBSITE из таблицы, нужно пытаться найти товар, у которого ACF event_ticket_url совпадает с WEBSITE после нормализации URL.
- Нормализация должна учитывать:
  - http/https;
  - завершающий слеш;
  - экранирование;
  - базовую унификацию query string, если это безопасно.

Приоритет 5. Поиск по составному ключу события
- Если надежные прямые источники не сработали, допускается fallback-поиск по комбинации:
  - RACE NAME / RACE NAME (PT)
  - EVENT START DATE
  - LOCATION (CITY)
  - CATEGORY / SUBCATEGORY
- Такой режим должен считаться менее надежным.

Приоритет 6. Поиск только по названию
- Разрешен только как последний fallback.
- Если найдено более одного кандидата, автоматическая запись запрещена.

8. Требования к восстановлению WP PRODUCT ID EN

8.1. Логика поиска EN product ID

Сначала попытаться:
1. извлечь ID из LINK RACEFINDER;
2. если есть URL с параметром p или post, взять его напрямую;
3. если есть публичный EN URL, получить ID из:
   - wp-json/wp/v2/product/{id}
   - shortlink
   - canonical/redirect chain;
4. если URL отсутствует, искать товар по WEBSITE;
5. если и это не сработало, искать по составному ключу.

8.2. Валидация кандидата EN

Перед записью WP PRODUCT ID EN нужно проверить:
- тип товара должен быть product;
- товар должен быть variable либо соответствовать логике проекта;
- дата события, если доступна в ACF, должна совпадать с EVENT START DATE;
- event_ticket_url, если доступен, должен совпадать или быть близким к WEBSITE;
- название должно быть похоже на ожидаемое.

Если проверка не пройдена, запись не выполнять.

9. Требования к восстановлению WP PRODUCT ID PT

9.1. Логика поиска PT product ID

Если найден EN product ID, PT нужно искать в следующем порядке:
1. по hreflang="pt-pt" на EN-странице товара;
2. по PT shortlink;
3. по PT wp-json/wp/v2/product/{id};
4. через явную WPML-связь, если есть доступный endpoint/маркер;
5. только если всё это не сработало, искать PT по составному ключу на PT-страницах.

9.2. Валидация кандидата PT

Перед записью WP PRODUCT ID PT нужно проверить:
- страница действительно PT;
- event_ticket_url совпадает с WEBSITE либо логически соответствует тому же событию;
- дата события совпадает;
- это перевод EN-карточки, а не другое событие.

Если PT найден неоднозначно, запись не выполнять.

10. Требования к восстановлению WP VARIATION ID EN/PT

10.1. Общий принцип

Вариации нельзя сопоставлять по сырым PT slug-значениям как по единственному ключу, потому что значения локализованы и могут отличаться:
- walking vs caminhada
- 5-km vs 5-km-pt
- 1000 vs 1000-pt

Нужно строить канонический нормализованный ключ вариации.

10.2. Источник вариаций

После нахождения product ID EN/PT необходимо получить список вариаций товара через публичный API, предпочтительно:
- wc/store/v1/products/{product_id}

Если этого API недостаточно, допускается использовать другой доступный WooCommerce endpoint, если он доступен в текущей конфигурации проекта.

10.3. Канонический ключ вариации

Для каждой строки вариации из таблицы и для каждой вариации товара EN/PT нужно строить единый канонический ключ.

В ключ должны входить только реально значимые атрибуты, присутствующие в данных:
- TYPE
- DISTANCE
- TEAM
- LICENSE
- RACE START DATE
- RACE START TIME
- ATTRIBUTE/VALUE, если они участвуют в вариативности

LOCATION и PRICE не являются основным ключом вариации:
- LOCATION обычно общий для товара;
- PRICE использовать только как дополнительную проверку, а не как основной идентификатор.

10.4. Правила нормализации

Для ключа необходима нормализация:

Нормализация типа/категории:
- Walking, caminhada -> walking
- Road Running, corrida-de-estrada -> road-running
- Half Marathon, meia-maratona -> half-marathon
- Kids Race, kids-race-pt -> kids-race
- Wheelchair race, wheelchair-race-pt -> wheelchair-race

Нормализация дистанции:
- 5 km, 5-km, 5-km-pt -> 5 km
- 10 km, 10-km, 10-km-pt -> 10 km
- 21,097 km, 21097-km, 21097-km-pt -> 21.097 km

Нормализация даты:
- 10/05/2026
- 10-05-2026
- 10-05-2026-pt
в единый формат:
- 2026-05-10

Нормализация времени:
- 10:00
- 1000
- 1000-pt
в единый формат:
- 10:00

Пустые значения:
- если атрибут отсутствует и это допустимый случай вариации, он должен нормализоваться консистентно как пустое значение;
- пример: kids variation без Distance.

10.5. Сопоставление вариаций

Алгоритм:
1. получить все вариации EN;
2. построить normalized_key для каждой EN variation;
3. получить все вариации PT;
4. построить normalized_key для каждой PT variation;
5. построить normalized_key для каждой дочерней строки таблицы;
6. найти точное совпадение по ключу:
   - в EN списке для WP VARIATION ID EN;
   - в PT списке для WP VARIATION ID PT.

10.6. Проверка уникальности

Если один normalized_key соответствует:
- более чем одной EN variation;
- более чем одной PT variation;
- либо более чем одной строке таблицы;
автоматическая запись запрещена, случай нужно пометить как ambiguous.

11. Что нужно записывать обратно в Google Sheets

При успешном восстановлении:
- WP PRODUCT ID EN
- WP PRODUCT ID PT
- WP VARIATION ID EN
- WP VARIATION ID PT

Запись должна быть точечной:
- обновлять только недостающие поля;
- уже заполненные значения не перезаписывать автоматически без отдельного флага принудительного режима.

12. Режимы запуска

Нужно предусмотреть минимум два режима:

12.1. Dry-run
- ничего не записывает в Google Sheets;
- только показывает, что было найдено;
- логирует:
  - найденные product IDs;
  - найденные variation IDs;
  - уровень уверенности;
  - неоднозначные случаи;
  - причины пропуска.

12.2. Apply
- записывает найденные значения в таблицу;
- записывает только те поля, которые прошли все проверки;
- не должен записывать значения при ambiguous-сопоставлении.

13. Требования к логированию

Для каждой основной строки нужно логировать:
- ID строки таблицы;
- название события;
- найден ли EN product ID;
- найден ли PT product ID;
- по какому источнику найден EN;
- по какому источнику найден PT;
- сколько вариаций найдено EN/PT;
- сколько дочерних строк вариаций удалось сопоставить;
- какие строки вариаций не удалось сопоставить;
- есть ли ambiguity;
- выполнена ли запись в таблицу или нет.

Причины отказа должны быть явными, например:
- no_product_match
- ambiguous_product_match
- pt_translation_not_found
- no_variation_match
- ambiguous_variation_match
- validation_failed

14. Требования к безопасности сопоставления

Скрипт должен быть консервативным.

Запрещено автоматически записывать ID, если:
- найдено несколько равноценных кандидатов;
- совпадение построено только по названию и нет дополнительных подтверждений;
- дата события не совпадает;
- WEBSITE явно не совпадает там, где он должен совпадать;
- вариация не имеет уникального normalized_key.

В сомнительных случаях нужно:
- вывести отчет;
- оставить поля без изменений;
- предложить ручную проверку.

15. Технические требования к коду

- Переиспользовать существующие утилиты проекта там, где это уместно.
- Не дублировать без необходимости код загрузки Google Sheets и конфигурации.
- Вынести нормализацию атрибутов вариаций в отдельные функции.
- Нормализацию URL и дат сделать отдельными тестируемыми функциями.
- Если понадобится словарь соответствий EN/PT значений, хранить его централизованно, а не вшивать фрагментарно по коду.

18.2. Matching tests

19. Ожидаемый пользовательский сценарий

Пользователь запускает recovery-скрипт.

Скрипт:
1. читает строки таблицы;
2. определяет, какие записи требуют восстановления ID;
3. находит EN/PT product IDs;
4. получает EN/PT variation IDs;
5. сопоставляет вариации по normalized_key;
6. пишет отчет;
7. в apply-режиме записывает найденные значения в Google Sheets.

После этого пользователь может перевести строку в Revised (complete), и основной pipeline должен обновить уже существующие товары и вариации, а не создавать дубликаты.

20. Критерии приемки

Задача считается выполненной, если:

1. Для старой опубликованной карточки, у которой известен хотя бы EN product URL или LINK RACEFINDER, скрипт может автоматически восстановить:
   - WP PRODUCT ID EN
   - WP PRODUCT ID PT

2. Для этой карточки скрипт может автоматически восстановить:
   - WP VARIATION ID EN
   - WP VARIATION ID PT
   по атрибутному сопоставлению, а не по названию вариации.

3. Сопоставление вариаций использует нормализованный канонический ключ, одинаковый для EN и PT.

4. Неоднозначные случаи не записываются автоматически.

5. Есть dry-run и apply режимы.

6. Добавлены тесты.

7. Обновлены документы:
   - docs/architecture.md
   - docs/requirements.md
   - docs/plan.md
   - README.md

8. Скрипт можно использовать повторно для массового восстановления старых данных без риска массовой неверной перезаписи.

21. Желательный дополнительный результат

Будет плюсом, если recovery-скрипт сможет формировать отдельный итоговый отчет:
- сколько строк обработано;
- сколько product IDs найдено;
- сколько PT-связей найдено;
- сколько variation IDs найдено;
- сколько строк требуют ручной проверки.

Формат отчета допускается:
- консольный summary;
- текстовый лог;
- либо отдельный CSV/TXT-файл.
